## 1. Project Summary

`tauri-codegen` is a fully automated multi-agent workflow that scaffolds a production-ready **Tauri v2 desktop application** from four user inputs. The generated project has three cleanly separated layers:

| Layer | Directory | Technology |
|---|---|---|
| Frontend (UI) | `frontend/` | React+Vite, Svelte+Vite, Blazor WASM, or NiceGUI |
| Backend (MCP server) | `backend/` | Node/Express, Python/FastAPI, or C#/ASP.NET Core |
| Rust core | `src-tauri/` | Tauri v2, tauri-plugin-shell, sidecar bridge |

Frontend ↔ Backend communication uses the **Model Context Protocol (MCP)** over one of three transports: `stdio` (sidecar), `http-sse`, or `streamable-http`. The orchestrator agent collects the four required inputs, then dispatches five specialist agents in sequence to build each layer, wiring them together at the end.

## 2. User Inputs Required

All four inputs are collected **upfront** by the orchestrator before any code generation begins.

| # | Parameter | Description | Allowed Values |
|---|---|---|---|
| 1 | `appName` | Application name, used for directory and product name | Any valid kebab-case string (e.g., `my-tauri-app`) |
| 2 | `frontendStack` | UI technology to generate | `react-vite` \| `svelte-vite` \| `blazor-wasm` \| `nicegui` |
| 3 | `backendStack` | Business logic and MCP server technology | `node-express` \| `python-fastapi` \| `csharp-aspnet` |
| 4 | `mcpTransport` | Wire protocol between frontend and backend | `stdio` \| `http-sse` \| `streamable-http` |

**Optional parameters** (collected with defaults):

| Parameter | Default | Description |
|---|---|---|
| `mcpTools` | `["query_data","run_script","get_system_info"]` | MCP tool names to scaffold |
| `packageManager` | `npm` | Root JS package manager (`npm` \| `pnpm` \| `yarn` \| `bun`) |
| `addLinting` | `true` | Scaffold ESLint / Clippy / dotnet-format configs |
| `addTesting` | `true` | Scaffold Vitest / pytest / xUnit test stubs |
| `tauriVersion` | `2` | Tauri major version (fixed at 2 for this workflow) |

## 3. Agent Roster

### 3.1 tauri-codegen.orchestrator

| Field | Value |
|---|---|
| **Role** | Entry point. Collects all user inputs via interactive prompt, writes `context.json`, invokes all sub-agents in order, and performs final integration validation. |
| **Model** | `claude-sonnet-4-5` (fallback: `gpt-4o`) |
| **Tools** | `Read`, `Write`, `Bash`, `Glob`, `Grep` |
| **Readonly** | `false` |
| **Inputs** | Raw user prompt (interactive or piped) |
| **Outputs** | `{appName}/context.json`, final validated project directory |

**Responsibilities**:
1. Parse or interactively collect `appName`, `frontendStack`, `backendStack`, `mcpTransport`, and optional params.
2. Validate input values against the allowed-values matrix (Section 7).
3. Write `{appName}/context.json` (shared state; see Section 5).
4. Invoke each sub-agent in the order defined in Section 4.
5. After all sub-agents complete, run final smoke-check: `npm run dev` dry-run and structural validation.
6. Report success or relay per-agent error summaries to the user.

### 3.2 tauri-codegen.scaffolder

| Field | Value |
|---|---|
| **Role** | Creates the root project directory skeleton, workspace manifest files, and empty placeholder directories. |
| **Model** | `claude-sonnet-4-5` |
| **Tools** | `Write`, `Bash`, `Read` |
| **Readonly** | `false` |
| **Inputs** | `context.json` → `appName`, `packageManager` |
| **Outputs** | `{appName}/` directory tree with root `package.json`, `Makefile`/`justfile`, `.gitignore`, `README.md`; empty `frontend/`, `backend/`, `src-tauri/` |

**Responsibilities**:
1. `mkdir -p {appName}/{frontend,backend,src-tauri/src,src-tauri/capabilities,src-tauri/binaries}`.
2. Write root `package.json` with npm workspaces (`["frontend","backend"]`) and `dev`/`build`/`tauri` scripts.
3. Write root `Makefile` with `dev`, `build`, `build-backend-sidecar`, `clean` targets (and optional `justfile` equivalent).
4. Write `.gitignore` covering `node_modules/`, `dist/`, `target/`, `__pycache__/`, `.env`.
5. Write skeleton `README.md` from `appName` and stack values in `context.json`.
6. **Validate**: all three directories exist; `package.json` is valid JSON.

### 3.3 tauri-codegen.frontend-generator

| Field | Value |
|---|---|
| **Role** | Generates the complete frontend layer for the chosen stack, including MCP client integration code. |
| **Model** | `claude-sonnet-4-5` |
| **Tools** | `Write`, `Read`, `Bash`, `Glob` |
| **Readonly** | `false` |
| **Inputs** | `context.json` → `appName`, `frontendStack`, `mcpTransport`, `packageManager`, `addLinting`, `addTesting` |
| **Outputs** | `frontend/` fully populated: stack source files, `frontend/src/mcp/client.ts` (or `.cs`/`.py`), `frontend/src/mcp/hooks.ts` (or store equivalent), `vite.config.ts`, `package.json` |

**Per-stack behaviour**:

| `frontendStack` | Scaffold command | MCP client file | Hook/store file |
|---|---|---|---|
| `react-vite` | `npm create vite@latest . -- --template react-ts` | `client.ts` (invoke or HTTP) | `hooks.ts` (`useMcpTool`) |
| `svelte-vite` | `npm create vite@latest . -- --template svelte-ts` | `client.ts` (invoke or HTTP) | `store.ts` (`createToolStore`) |
| `blazor-wasm` | `dotnet new blazorwasm -n Frontend` | `McpClientService.cs` | `Program.cs` DI registration |
| `nicegui` | `python -m venv .venv && pip install nicegui mcp[cli]` | `main.py` (inline async MCP client) | N/A (NiceGUI handles UI) |

**MCP client variant selection**:
- `mcpTransport == stdio` → write the `invoke()`-based client (Tauri IPC bridge); not available for `blazor-wasm`.
- `mcpTransport == http-sse` → write the `SSEClientTransport` client.
- `mcpTransport == streamable-http` → write the `StreamableHTTPClientTransport` client.

**Validate**:
- `frontend/src/mcp/client.ts` (or equivalent) exists and exports `callMcpTool`.
- `npm run dev` in `frontend/` exits `0` within a 30-second timeout.

### 3.4 tauri-codegen.backend-generator

| Field | Value |
|---|---|
| **Role** | Generates the complete backend MCP server for the chosen stack, with example tool implementations. |
| **Model** | `claude-sonnet-4-5` |
| **Tools** | `Write`, `Read`, `Bash`, `Glob` |
| **Readonly** | `false` |
| **Inputs** | `context.json` → `appName`, `backendStack`, `mcpTransport`, `mcpTools`, `addLinting`, `addTesting` |
| **Outputs** | `backend/` fully populated: server bootstrap, one tool file per entry in `mcpTools`, package manifest, health endpoint |

**Per-stack behaviour**:

| `backendStack` | Entry point | Tool directory | Package manifest |
|---|---|---|---|
| `node-express` | `backend/src/server.ts` + `stdio-server.ts` | `backend/src/tools/` | `package.json` + `tsconfig.json` |
| `python-fastapi` | `backend/main.py` | `backend/tools/` | `pyproject.toml` |
| `csharp-aspnet` | `backend/Backend/Program.cs` | `backend/Backend/Tools/` | `Backend.csproj` |

**Default tools scaffolded** (if `mcpTools` is empty, use this set):
- `query_data` — query structured data; inputs: `query: string`, `limit?: number`
- `run_script` — execute a sandboxed shell command; inputs: `script: string`, `timeout?: number`
- `get_system_info` — return OS/hardware metadata; inputs: none

**Validate**:
- `backend/` starts without error (test with `npm run dev`, `python main.py --check`, or `dotnet build`).
- HTTP transports: `GET /health` returns `{"status":"ok"}`.
- MCP `tools/list` response includes all declared tools.

### 3.5 tauri-codegen.tauri-core-generator

| Field | Value |
|---|---|
| **Role** | Generates the complete `src-tauri/` Rust core: `Cargo.toml`, `tauri.conf.json`, `main.rs`, `lib.rs`, capabilities, and sidecar config. |
| **Model** | `claude-sonnet-4-5` |
| **Tools** | `Write`, `Read`, `Bash`, `Glob` |
| **Readonly** | `false` |
| **Inputs** | `context.json` → `appName`, `frontendStack`, `backendStack`, `mcpTransport` |
| **Outputs** | `src-tauri/Cargo.toml`, `src-tauri/tauri.conf.json`, `src-tauri/src/main.rs`, `src-tauri/src/lib.rs`, `src-tauri/capabilities/default.json` |

**Key generation rules**:

| `mcpTransport` | `tauri.conf.json` → `bundle.externalBin` | `lib.rs` transport method | `capabilities` extra permissions |
|---|---|---|---|
| `stdio` | `["binaries/backend"]` | stdin/stdout JSON-RPC bridge | `shell:allow-spawn`, scoped to `backend` binary |
| `http-sse` | _(omit)_ | HTTP client request to `localhost:3001` | `shell:allow-open` only |
| `streamable-http` | _(omit)_ | HTTP POST to `localhost:3001/mcp` | `shell:allow-open` only |

**Sidecar state management** (stdio only): scaffold `BackendState(Mutex<Option<CommandChild>>)` with `start_backend` / `stop_backend` Tauri commands.

**`tauri.conf.json` fields populated from `context.json`**:
- `productName` ← `appName`
- `identifier` ← `com.example.{appName}`
- `build.devUrl` ← stack-specific (e.g., `http://localhost:5173` for Vite stacks, `http://localhost:8080` for NiceGUI)
- `build.frontendDist` ← `../frontend/dist` (Vite stacks) or `../frontend/Frontend/wwwroot` (Blazor)

**Validate**:
- `cargo build --manifest-path src-tauri/Cargo.toml` exits `0`.
- All Tauri commands are registered in `invoke_handler!`.
- `capabilities/default.json` grants at minimum `core:default`.

### 3.6 tauri-codegen.wirer

| Field | Value |
|---|---|
| **Role** | Wires all layers together: reconciles `tauri.conf.json` paths, configures CORS and sidecar env vars, writes final `Makefile`/`justfile` with complete `dev`/`build`/`test` targets, validates end-to-end structure. |
| **Model** | `claude-sonnet-4-5` |
| **Tools** | `Read`, `Write`, `Bash`, `Glob`, `Grep` |
| **Readonly** | `false` |
| **Inputs** | `context.json`, entire generated project tree |
| **Outputs** | Updated `src-tauri/tauri.conf.json`, updated root `Makefile`/`justfile`, `frontend/src/config.ts` (env var shim), backend CORS configuration, structural validation report written to `{appName}/.codegen-validation.json` |

**Wiring tasks**:
1. Verify `tauri.conf.json` `devUrl` and `frontendDist` match the actual frontend stack output paths.
2. For `stdio` transport: confirm `bundle.externalBin` contains `binaries/backend`; add `build-backend-sidecar` target to `Makefile`.
3. For HTTP transports: add `beforeDevCommand` in `tauri.conf.json` to start both `frontend` and `backend` concurrently.
4. Inject `CORS` origin allowlist (`http://localhost:5173`, `tauri://localhost`) into backend server bootstrap.
5. Write `frontend/src/config.ts` with `BACKEND_URL` and `MCP_TRANSPORT` env-var shim.
6. Finalize root `Makefile` targets:
   - `make dev` → starts all three layers
   - `make build` → builds frontend, backend binary, Tauri bundle
   - `make test` → runs frontend, backend, Rust, and MCP validation tests
   - `make clean` → removes `dist/`, `target/`, `__pycache__/`
7. Run structural validation: assert all required files exist, write `.codegen-validation.json`.

**Validate**:
- `npm run dev` from workspace root starts without errors (dry-run with `--dry-run` flag or a 5-second test).
- `.codegen-validation.json` contains no `"status": "fail"` entries.

## 4. Execution Flow

Agents execute **sequentially**. Each agent receives `context.json` plus the outputs described in the handoff notes below.

```
User Prompt
    │
    ▼
[1] tauri-codegen.orchestrator
    │  Writes: context.json
    │
    ▼
[2] tauri-codegen.scaffolder
    │  Reads:  context.json
    │  Writes: {appName}/ skeleton (dirs + root configs)
    │
    ▼
[3] tauri-codegen.frontend-generator
    │  Reads:  context.json, frontend/ (empty)
    │  Writes: frontend/ (full source + MCP client)
    │
    ▼
[4] tauri-codegen.backend-generator
    │  Reads:  context.json, backend/ (empty)
    │  Writes: backend/ (full source + MCP server + tools)
    │
    ▼
[5] tauri-codegen.tauri-core-generator
    │  Reads:  context.json, frontend/package.json (for devUrl),
    │          backend/package.json or pyproject.toml (for sidecar cmd)
    │  Writes: src-tauri/ (Cargo.toml, tauri.conf.json, lib.rs, capabilities/)
    │
    ▼
[6] tauri-codegen.wirer
    │  Reads:  context.json, ALL generated files
    │  Writes: reconciled tauri.conf.json, final Makefile, config.ts, .codegen-validation.json
    │
    ▼
Orchestrator: final smoke-check + user report
```

**Parallelism note**: Steps 3 and 4 (frontend-generator and backend-generator) are logically independent and **may be run in parallel** by an orchestrator that supports concurrent tool dispatch. Steps 5 and 6 must wait for both to complete.

## 5. Context Passing Strategy

All agents share state through a single **`context.json`** file written by the orchestrator at the root of the generated project directory before any sub-agent is invoked.

### Schema

```json
{
  "appName": "my-tauri-app",
  "frontendStack": "react-vite",
  "backendStack": "node-express",
  "mcpTransport": "streamable-http",
  "mcpTools": ["query_data", "run_script", "get_system_info"],
  "packageManager": "npm",
  "addLinting": true,
  "addTesting": true,
  "tauriVersion": 2,
  "generatedAt": "2026-04-19T00:00:00Z",
  "agents": {
    "scaffolder":          { "status": "pending" },
    "frontend-generator":  { "status": "pending" },
    "backend-generator":   { "status": "pending" },
    "tauri-core-generator":{ "status": "pending" },
    "wirer":               { "status": "pending" }
  }
}
```

### Rules

1. **Orchestrator writes** `context.json` once; sub-agents **read** it at startup.
2. Each sub-agent **updates its own status** entry (`"running"` → `"complete"` or `"failed"`) before exiting.
3. The wirer additionally writes `"wiringNotes"` key with a list of reconciliation actions taken, for auditability.
4. The orchestrator reads the final `context.json` to verify all statuses are `"complete"` before declaring success.
5. `context.json` is written to `{appName}/context.json` (inside the generated project) so the file travels with the generated code.

## 6. Error Handling

### Per-Agent Failure Modes and Recovery

| Agent | Likely Failure | Recovery Strategy |
|---|---|---|
| **orchestrator** | Invalid input values (e.g., unsupported `frontendStack`) | Validate against the stack matrix (Section 7) before writing `context.json`; surface a clear error listing allowed values. Re-prompt the user for the invalid field only. |
| **scaffolder** | Disk permission error; directory already exists | Check for existing directory before creation. If `{appName}/` already exists, abort with `"Directory exists — use a different appName or delete it first."` |
| **frontend-generator** | Vite scaffold CLI failure (network timeout, wrong Node version); `blazor-wasm` requires .NET SDK absent | Pre-check required tools (`node --version`, `dotnet --version`, `python --version`) before running scaffold commands. On failure, write `context.json` status `"failed"` with `"error"` key and halt; orchestrator surfaces the error. |
| **backend-generator** | Missing runtime (Python venv, `dotnet` SDK, Node); npm install failure | Same pre-check strategy as frontend-generator. For Python, verify `python3 -m venv` works. For C#, verify `dotnet new`. |
| **tauri-core-generator** | Rust / Cargo not installed; `cargo build` compilation error (e.g., wrong Tauri feature flags) | Pre-check `rustc --version` and `cargo --version`. On `cargo build` failure, capture stderr and write it to `{appName}/tauri-build-errors.log`; surface a truncated summary to the orchestrator. |
| **wirer** | Missing output files from a prior agent (agent left `"failed"` status); `tauri.conf.json` path mismatch | Read each prior agent's status from `context.json` before beginning. If any is not `"complete"`, abort and report which agent failed. For path mismatches, use `Glob` to discover actual output paths and reconcile rather than fail. |

### Global Recovery Policy

- Any agent that sets its status to `"failed"` causes the orchestrator to **halt the pipeline** and report the failure clearly to the user.
- The orchestrator does **not** attempt to re-run a failed agent automatically (risk of partial state corruption). It instructs the user to fix the pre-condition and re-run the specific agent step.
- Partial output from a failed agent is left in place (not cleaned up) to aid debugging.

## 7. Stack Matrix

### Supported Combinations

✅ = supported | ⚠️ = supported with caveat | ❌ = not supported

| `frontendStack` | `backendStack` | `stdio` | `http-sse` | `streamable-http` |
|---|---|:---:|:---:|:---:|
| `react-vite` | `node-express` | ✅ | ✅ | ✅ |
| `react-vite` | `python-fastapi` | ✅ | ✅ | ✅ |
| `react-vite` | `csharp-aspnet` | ✅ | ✅ | ✅ |
| `svelte-vite` | `node-express` | ✅ | ✅ | ✅ |
| `svelte-vite` | `python-fastapi` | ✅ | ✅ | ✅ |
| `svelte-vite` | `csharp-aspnet` | ✅ | ✅ | ✅ |
| `blazor-wasm` | `node-express` | ❌ | ✅ | ✅ |
| `blazor-wasm` | `python-fastapi` | ❌ | ✅ | ✅ |
| `blazor-wasm` | `csharp-aspnet` | ❌ | ✅ | ✅ |
| `nicegui` | `node-express` | ✅ | ✅ | ⚠️ |
| `nicegui` | `python-fastapi` | ✅ | ✅ | ⚠️ |
| `nicegui` | `csharp-aspnet` | ✅ | ✅ | ⚠️ |

**Caveats**:
- `blazor-wasm` + `stdio`: Blazor WASM runs in the browser sandbox and cannot use the Tauri IPC bridge directly. HTTP transport required.
- `nicegui` + `streamable-http`: NiceGUI's Python `mcp` SDK supports Streamable HTTP via `starlette`, but the generated integration code requires the `mcp>=1.6` package. Flag as experimental.
- `nicegui` uses `http://localhost:8080` as `devUrl` (not `5173`); `tauri.conf.json` must be set accordingly by tauri-core-generator.

| `blazor-wasm` | `5000` | `http://localhost:5000` |
